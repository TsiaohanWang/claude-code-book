# 附录 E：工程检查清单

> 本附录提供五套可执行的检查清单，覆盖 Agent Harness 从开发到生产的完整生命周期。每套清单都可以直接复制到你的项目管理工具中使用。

---

## E.1 运行前检查清单

在启动 Agent 会话之前，确认以下项目：

### 环境检查

- [ ] Node.js 版本 ≥ 18（`node --version`）
- [ ] Git 版本 ≥ 2.30（`git --version`）
- [ ] 网络连接正常（能访问 Anthropic API）
- [ ] API 密钥已配置且有效
- [ ] 工作目录存在且有读写权限

### 权限配置检查

- [ ] 权限模式已明确选择（default/auto/bypass）
- [ ] 高风险工具（Bash、FileWrite）的 deny/ask 规则已配置
- [ ] 项目特定的规则已写入 CLAUDE.md
- [ ] 敏感文件/目录已加入 deny 规则（.env、credentials、secrets）

### MCP 服务器检查

- [ ] 已配置的 MCP 服务器可正常连接
- [ ] MCP 工具的权限规则已配置
- [ ] 未知来源的 MCP 服务器已审查安全性

---

## E.2 Agent Harness 设计评审清单

在设计或评审一个 Agent Harness 架构时，逐项检查：

### 核心循环

- [ ] 对话循环使用 AsyncGenerator（而非简单 while + callback）
- [ ] 循环状态使用不可变对象（每次 continue 创建新 State）
- [ ] `transition` 字段记录每次状态转换的原因
- [ ] 依赖注入使核心循环可测试（QueryDeps 接口）
- [ ] 终止原因有细粒度分类（≥ 8 种）

### 工具系统

- [ ] 工具定义遵循统一接口（名称、Schema、权限、执行、UI）
- [ ] 使用 Zod Schema 做运行时参数验证
- [ ] 工具注册支持条件注册和延迟加载
- [ ] 并发分区算法根据 `isConcurrencySafe()` 调度
- [ ] 每个工具都有完整的 UI 渲染方法（6 个生命周期方法）

### 权限系统

- [ ] 权限检查为四阶段管线（validateInput → 规则匹配 → 上下文评估 → 交互确认）
- [ ] deny 规则优先级高于 allow（deny > ask > allow）
- [ ] 权限规则支持多来源优先级（session > cli > project > user）
- [ ] 高风险工具（Bash）有额外的上下文评估逻辑
- [ ] 权限决策可持久化（避免重复询问用户）

## E.3 上下文治理配置清单

### 窗口计算

- [ ] 有效窗口 = 模型窗口 - min(最大输出token, 20000)
- [ ] 警告阈值 = 有效窗口 - 20000 tokens
- [ ] 自动压缩阈值 = 有效窗口 - 13000 tokens
- [ ] 阻塞阈值 = 有效窗口 - 3000 tokens

### 压缩策略

- [ ] 五级压缩已配置：Snip → Microcompact → Context Collapse → AutoCompact
- [ ] 压缩顺序从轻量到重量排列
- [ ] 自动压缩断路器阈值 = 3（`MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES`）
- [ ] 压缩自身的 `prompt_too_long` 有回退处理（`truncateHeadForPTLRetry`）

### 记忆系统

- [ ] 记忆类型为闭合四类型（user/feedback/project/reference）
- [ ] MEMORY.md 索引已初始化
- [ ] 只保存不可从项目状态推导的信息
- [ ] 记忆文件有合理的大小限制

---

## E.4 安全加固清单

### 输入安全

- [ ] 所有工具输入经过 Zod Schema 验证
- [ ] 文件路径经过规范化和白名单检查
- [ ] Bash 命令经过风险评估和模式匹配
- [ ] 外部数据（API 响应、MCP 结果）不直接信任

### 执行安全

- [ ] 高风险操作（文件删除、Git push、npm publish）需要用户确认
- [ ] 工具执行有超时限制
- [ ] 输出大小有上限（防止上下文爆炸）
- [ ] 子智能体有独立的权限范围

### 状态安全

- [ ] 状态更新使用不可变模式（整体替换而非字段修改）
- [ ] 子智能体不能修改父智能体的状态
- [ ] 中断处理有账本闭合（每个 tool_use 都有对应 tool_result）
- [ ] 敏感信息不出现在日志或记忆中

### 供应链安全

- [ ] MCP 服务器来源已审查
- [ ] 钩子脚本来源已审查
- [ ] 技能文件来源已审查
- [ ] 第三方工具的权限范围已限制

---

## E.5 生产部署清单

### 可观测性

- [ ] 对话循环的每次状态转换都有日志
- [ ] 工具执行的成功/失败/耗时有指标
- [ ] 上下文使用量有监控（token 使用率）
- [ ] 自动压缩的触发和结果有记录
- [ ] 断路器的状态变化有告警

### 容错性

- [ ] API 调用失败有重试机制（指数退避）
- [ ] `prompt_too_long` 有分层恢复策略
- [ ] `max_output_tokens` 截断有延续优先恢复
- [ ] 自动压缩有断路器保护
- [ ] 中断处理有账本闭合

### 性能

- [ ] 启动时间 < 2 秒（延迟加载、并行预取）
- [ ] 工具并发执行（安全的工具并行调度）
- [ ] 缓存感知的 Prompt 构建（命中 API 侧缓存）
- [ ] 流式输出（用户无需等待完整响应）

### 团队协作

- [ ] CLAUDE.md 已纳入版本控制
- [ ] 权限配置已纳入版本控制
- [ ] 有规则评审流程（PR 审查）
- [ ] 有失败编码机制（从错误中学习并更新规则）
- [ ] 新成员可以独立使用 Agent 而无需口头辅导
