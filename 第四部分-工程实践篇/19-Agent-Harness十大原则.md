# 第 19 章 Agent Harness 十大原则

> *"吾道一以贯之。"* ——《论语·里仁》

**阅读指南：** 本章是全书的总结与提炼。十大原则从前面 17 章的具体分析中抽象而来，每条原则都附有来源章节、Claude Code 实现位置和可迁移应用场景。如果你时间有限，只读这一章也能获得全书 80% 的核心认知。

---

## 为什么需要原则？

具体的技术会过时——Bun 可能被取代，Zod 可能被更好的库替代，甚至 Claude Code 本身也可能被新产品超越。但**设计原则**不会过时。它们是从具体实践中提炼出的通用智慧，可以迁移到任何 Agent 系统。

用一个比喻来说：具体技术是"鱼"，设计原则是"渔"。授人以鱼不如授人以渔——理解了原则，你就能在面对新场景时做出一致的设计决策，即使那个场景是 Claude Code 从未遇到过的。

---

## 原则一：约束执行是核心能力

> **The key capability of an agent system is constrained execution.**

Agent 的核心能力不是"聪明"——模型的聪明是 LLM 提供的。Agent 的核心能力是**约束**：确保模型的"聪明"被安全、可靠、高效地转化为实际操作。

一个没有约束的 Agent 就像一个没有刹车的赛车——速度越快，危险越大。约束不是限制能力，而是让能力变得可用。

**Claude Code 实现位置：**
- 四阶段权限管线（第 4 章）
- 工具调度纪律：并发分区、串行执行（第 3 章）
- 高风险工具的高密度约束：BashTool 的 prompt 规则（第 3 章）

**可迁移场景：** 任何涉及 LLM 与外部世界交互的系统——自动化运维、数据处理管道、CI/CD 集成。

---

## 原则二：Prompt 是控制平面，不是人格装饰

> **Prompt is part of the control plane, not personality decoration.**

许多人仍然把 Prompt 当作"给 AI 设定人设"的文本。但在 Agent 系统中，Prompt 是**行为控制平面**——它定义执行边界、失败行为和报告责任。

Claude Code 的系统 Prompt 不是"你是一个有帮助的助手"，而是分层组装的控制指令：默认规则 → 项目规则 → 自定义规则 → Agent 角色 → 记忆注入 → 输出风格。每一层都有明确的优先级和覆盖关系。

**Claude Code 实现位置：**
- `getSystemPrompt()` 的分层组装逻辑（第 1 章）
- CLAUDE.md 的就近优先级（第 5 章、第 17 章）
- 系统 Prompt 的缓存感知构建（第 1 章）

**可迁移场景：** 任何使用 LLM 的系统——不要把 Prompt 当作彩蛋，把它当作架构的一部分来设计。

---

## 原则三：错误是主路径的一部分

> **Errors are part of the main path, not exceptions to it.**

传统软件系统把错误当作"异常"——happy path 是正文，catch 块是脚注。Agent 系统不能这样做。`prompt_too_long`、`max_output_tokens` 截断、工具执行失败、用户中断——这些不是边缘情况，而是结构性常态。

一个成熟的 Agent 系统不是"从不犯错"的系统，而是"犯错后知道如何恢复"的系统。

**Claude Code 实现位置：**
- withheld 机制：先恢复，再报告（第 8 章）
- 分层恢复策略：折叠 → 响应式压缩 → 表面化（第 8 章）
- 断路器模式：防止恢复变成死循环（第 8 章）
- 十种终止原因的精细划分（第 2 章）

**可迁移场景：** 任何长时间运行的自动化系统——错误处理不是事后补充，而是核心设计。

---

## 原则四：工具调用必须服从调度纪律

> **Tool calls must obey scheduling discipline.**

当模型可以调用工具时，风险从"修辞风险"（说错话）转变为"执行风险"（做错事）。关键问题是：**谁决定工具如何运行？**

Claude Code 的答案是：运行时决定。根据工具的 `isConcurrencySafe()` 属性决定并行还是串行执行；根据 `isReadOnly()` 属性决定是否需要权限检查；根据 `isDestructive()` 属性决定风险等级。

工具不是模型能力的自然延伸，而是需要调度纪律的受管执行单元。

**Claude Code 实现位置：**
- `partitionToolCalls()` 并发分区算法（第 3 章）
- `StreamingToolExecutor` 四阶段状态机（第 3 章）
- 上下文修改器的缓冲与重放（第 3 章）

**可迁移场景：** 任何多工具编排系统——不要让模型直接决定执行顺序，让运行时根据工具属性调度。

---

## 原则五：高风险能力需要高密度约束

> **The stronger the capability, the finer the control.**

在所有工具中，Bash 是最不可信的——它几乎没有领域边界：可以直接操作文件、进程、网络和 Git 仓库，还有 shell 重定向和管道语句叠加。任何过度信任 Bash 的系统通常都会有实际后果。

Claude Code 的做法是：为 Bash 工具配备极其详细的 prompt 规则——不要随意修改 git config、不要跳过 hooks、不要随便 `git add .`、不要在 pre-commit 失败后使用 `--amend`、不要在未被明确要求时提交代码、不要默认 push。

这不是过度设计，而是**高风险接口需要高密度约束**的基本原则。

**Claude Code 实现位置：**
- `BashTool/prompt.ts` 中的详细操作规则（第 3 章）
- 四阶段权限管线中的上下文评估（第 4 章）
- Bash 规则匹配的优先级排序（第 4 章）

**可迁移场景：** 任何涉及文件系统操作、数据库修改、网络请求的工具——能力越强，约束越细。

---

## 原则六：恢复必须分层、可熔断

> **Layer recovery paths; make them countable, rate-limited, and breakable.**

恢复不是一个"尝试或不尝试"的二元选择。好的恢复系统应该：

1. **分层**：先尝试代价最小的方案（flush 折叠），再升级到代价更大的方案（响应式压缩），最后才表面化错误。
2. **可计数**：记录每次恢复尝试的结果。
3. **可限速**：限制单位时间内的恢复尝试次数。
4. **可熔断**：连续失败达到阈值后停止尝试。

一个没有刹车的恢复系统就像一辆没有刹车的汽车——它不是在恢复，而是在加速。

**Claude Code 实现位置：**
- `hasAttemptedReactiveCompact` 防死循环守卫（第 8 章）
- `MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3` 断路器阈值（第 8 章）
- `truncateHeadForPTLRetry()` 压缩自身的恢复（第 8 章）
- `max_output_tokens` 的两阶段恢复（第 8 章）

**可迁移场景：** 任何有自动重试/恢复机制的系统——恢复逻辑必须有自己的治理机制。

---

## 原则七：上下文治理是预算制度，不是截断操作

> **Context governance is a budgeting regime, not a truncation operation.**

上下文窗口是 Agent 最珍贵的资源。管理它的方式不应该是"满了就截断"，而应该是一套精细的预算制度：

- 安全区域（0-85%）：正常工作
- 警告区域（85-90%）：提醒用户注意
- 危险区域（90-95%）：自动压缩触发
- 阻塞区域（95-100%）：拒绝新请求

五级压缩策略（Tool Result 预算裁剪 → Snip → Microcompact → Context Collapse → AutoCompact）从轻量到重量排列，每一步都先尝试最小代价的方案。

**Claude Code 实现位置：**
- 有效窗口公式和阈值常量（第 7 章）
- 五级渐进压缩管线（第 7 章、第 2 章）
- 缓存感知的压缩策略（第 7 章）

**可迁移场景：** 任何有资源限制的系统——不要等到资源耗尽才处理，建立预警和分级响应机制。

---

## 原则八：记忆只保存不可推导的信息

> **Memory stores only what cannot be derived from the current state.**

代码模式、架构、文件结构和 Git 历史都可以通过工具实时获取——这些不属于记忆。记忆应该只保存**不可从当前项目状态推导的信息**：用户的角色和偏好、被验证过的做法、项目的决策背景、外部参考链接。

这个原则防止了记忆系统的膨胀和过时。如果信息可以从项目状态推导，它就不应该被记忆——因为项目状态变化时，记忆中的副本会变成过时的"幻影"。

**Claude Code 实现位置：**
- 四种闭合记忆类型：user / feedback / project / reference（第 6 章）
- MEMORY.md 索引的只保存元数据策略（第 6 章）
- Fork 模式的记忆继承机制（第 6 章）

**可迁移场景：** 任何有持久化存储的系统——不是所有信息都值得保存，只保存无法推导的部分。

---

## 原则九：多智能体工作与验证必须分离

> **Multi-agent work and verification must not be blended into one vague mechanism.**

当系统从"一个 Agent 自救"演进到"一个 Agent 委托、另一个验证"时，错误和恢复不再是纯粹的单线程问题，而变成了组织设计问题。

Claude Code 的协调器模式遵循"只编排不执行"的约束：Coordinator 负责分发任务和汇总结果，但不直接执行工具。验证独立于实现——验证者 ≠ 实现者。

这种分离确保了：实现者的偏见不会污染验证结果，验证者的反馈可以被实现者独立处理。

**Claude Code 实现位置：**
- 协调器模式的双重门控（第 11 章）
- 子智能体的独立上下文和权限范围（第 10 章）
- Fork 模式的上下文继承与隔离（第 10 章）

**可迁移场景：** 任何涉及代码审查、测试、审计的系统——执行者和验证者应该是不同的角色。

---

## 原则十：制度比聪明更可靠

> **Structure is less flashy than smartness, but usually more reliable.**

模型会犯错。工具会放大后果。上下文会膨胀。状态会污染后续轮次。用户会中断。失败会重复。

面对这些现实，系统不能靠"聪明"来维持秩序，而必须靠**结构**。结构不如聪明耀眼，但通常更可靠。

**Claude Code 实现位置：**
- 不可变状态流转（第 1 章、第 2 章）
- 依赖注入使测试成为可能（第 2 章）
- 类型系统强制的接口契约（第 3 章）
- 四阶段权限管线的纵深防御（第 4 章）

**可迁移场景：** 一切——不要相信"更强的模型会解决系统问题"的幻想。模型变强只会让工具调用更多、执行链更长、失败后果更严重。工程结构是唯一可靠的秩序来源。

## 结语

这十条原则不是教条——它们是从 Claude Code 的具体实践中提炼出的通用智慧。当你在自己的项目中遇到 Claude Code 没有覆盖的场景时，回到这些原则，问自己：

- 我的系统有足够的约束吗？（原则 1）
- 我的 Prompt 是在控制行为还是在装饰人格？（原则 2）
- 我的错误处理是主路径的一部分吗？（原则 3）
- 我的工具有调度纪律吗？（原则 4）
- 高风险操作有足够的约束吗？（原则 5）
- 我的恢复逻辑有刹车吗？（原则 6）
- 我的上下文管理是预算制度还是暴力截断？（原则 7）
- 我的记忆是否只保存了不可推导的信息？（原则 8）
- 执行和验证是否分离？（原则 9）
- 我是在靠聪明还是靠结构？（原则 10）

回答好这十个问题，你就拥有了一套可迁移到任何 Agent 系统的设计检验清单。
